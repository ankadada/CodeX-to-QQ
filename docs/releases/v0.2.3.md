# CodeX-to-QQ v0.2.3

这是一版围绕“多 provider 兼容”和“公开协作可理解性”的更新：Claude Code 现在可以作为可选 provider 接进来，但默认行为仍然保持 Codex，不会打断现有用户。

## 这版更新了什么

### 1. 新增 Claude Code provider

现在可以在 `.env` 里显式开启：

```bash
PROVIDER=claude
CLAUDE_BIN=claude
```

如果不设置，默认仍然走 Codex。

这意味着：

- 原来的 Codex 用户可以无痛升级
- 想切到 Claude Code 的用户不需要再维护另一套 QQ bridge
- 上游仓库也不会因为引入 Claude 支持而变成 breaking change

### 2. Codex / Claude 状态和 workspace 彻底隔离

这版把以下内容都做成了 provider 级别隔离：

- state 文件
- lock 文件
- 默认 workspace

这样一来，Codex 和 Claude 不会再共享同一套默认运行上下文，避免：

- 会话串线
- workspace 污染
- 状态文件互相覆盖

同时，对旧版的 `data/state.json` 也保留了兼容读取路径，升级后不会直接丢会话信息。

### 3. Claude 的附件和 token 统计更稳

这次没有强行假设 Claude CLI 支持 Codex 那套图片参数，而是改成：

- 让 Claude 通过 workspace 里的 `local_path` 读取附件
- 对图片继续保留 OCR 文本预览辅助
- 对 token usage 做更宽松的字段解析

这样更贴近 Claude Code 当前真实 CLI 行为，也更利于后续兼容不同 usage 结构。

### 4. `doctor` / 文档 / 测试全部跟上 provider 变化

现在：

- `doctor` 会显示当前 provider
- 会检查对应的 CLI binary
- 会识别 provider 专属的 state 文件
- 中英文 README 都写明了 Claude 的开启方式
- 测试已覆盖新的 provider usage 解析与兼容逻辑

## 适合升级的人

这版特别适合：

- 已经长期用 CodeX-to-QQ 的 Codex 用户
- 想试 Claude Code，但不想另开一套机器人服务的人
- 准备把仓库公开给更多人一起协作、提 issue、提交 PR 的维护者

## 升级方式

```bash
git pull
npm install
npm run ci
```

如果你准备切到 Claude Code，再把 `.env` 改成：

```bash
PROVIDER=claude
CLAUDE_BIN=claude
```

后台服务用户升级后再执行：

```bash
npm run service:restart
```

## 说明

- 这版仍然默认面向自建、可信使用场景
- `dangerous` 模式的安全边界没有变化
- Claude 支持是 opt-in，不会影响默认 Codex 路径
